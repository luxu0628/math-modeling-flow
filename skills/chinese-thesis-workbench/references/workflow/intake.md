# Intake Prompt

用于锁定用户输入约束。

硬约束：

- 第一次回复时，先收资料，不开写初稿。
- 若用户尚未提供模板、样文、任务书、开题报告等辅助资料，必须优先索要，除非用户明确说“没有”或“按项目自己写”。
- 如果已经给了样文或模板，必须先分析，再回传目录、字数和版式观察，不得跳过分析直接写正文。
- 输入优先级默认是：学校模板 > 任务书/开题报告 > 用户明确要求 > 样文 > 项目源码 > README > 旧说明文档。
- 旧部署说明、历史项目介绍、演示文档默认视为低可信来源。
- 每一项材料都必须在 `paper-context/workflow/material-inventory.md` 中标记为 `required`、`strongly_recommended` 或 `optional`，并写清缺失影响、是否可有限继续、用户下一步。

必须确认：

1. 论文类型
2. 课题名称
3. 是否有学校模板
4. 是否有往届样文
5. 是否有字数要求
6. 是否必须交付 `.docx`
7. 是否有开题报告、任务书或封面字段要求
8. 如果需要真实页面截图，系统访问地址或启动方式是什么
9. 主论文文件名是否必须使用论文标题
10. 是否需要额外附件 `.docx`

必须选择交付路径：

1. 从零生成（默认样式）
2. 从零生成（按样文版式贴近）
3. 学校模板副本填充（推荐用于格式要求严格的学校）
4. 在已有初稿上做局部增删改

选择结果必须记录到 `paper-context/workflow/user-decisions.md`。

首次回复时，必须明确告诉用户可以直接发送本地路径，示例：

- `D:\论文模板.docx`
- `D:\历届样文.pdf`
- `D:\开题报告.docx`

如果用户一次没给全，不要连续抛很多问题。优先确认：

1. 是否有模板或样文
2. 是否有开题报告或任务书
3. 是否有字数要求
4. 是否最终要 Word
5. 如果需要截图，系统如何打开
6. 论文标题是什么

材料不全时，不要只说“请补材料”。必须告诉用户：

1. 缺失材料影响哪些章节或交付物
2. 当前是否可以有限继续
3. 有限继续会牺牲什么
4. 用户可以选择补充、确认没有、或暂缓
5. 推荐路径是什么

拿到模板、样文或开题报告后，不要立刻开写，必须先分析。这里的分析只用于目录、字数、版式观察和图表节奏；解析结果不得直接驱动 DOCX 排版，除非用户明确选择样文版式贴近生成或学校模板副本填充：

1. 目录结构是否完整
2. 各章大致字数
3. 标题、正文、摘要、关键词、图题、表题、表格内容的版式观察
4. 图、表、代码通常出现在什么章节

如果样文或模板是 `.docx`，优先执行解析脚本或接收用户/外部工具提供的等价解析结果：

- `python scripts/docx/analyze_docx.py <sample.docx> --json-out paper-context/workflow/sample-docx-analysis.json`

如果样文是 PDF，优先执行解析脚本或接收用户/外部工具提供的等价解析结果：

- `python scripts/docx/analyze_sample_pdf.py <sample.pdf> --json-out paper-context/workflow/sample-analysis.json`

解析脚本只是轻量分析输入层。若脚本输出不足、失败或只能给出弱结构结果，不要把脚本问题当作论文流程能力已经完成；应在样文/模板分析报告中标记 `partial`、`failed` 或 `needs_confirmation`，并继续生成有限的用户可读分析。解析结果不得直接作为 DOCX 排版规则，除非用户明确选择 `--sample-analysis` 样文版式贴近生成或 `--from-template` 模板副本填充。

分析完成后，必须先回给用户：

1. 输入资料表
2. 当前建议目录表
3. 各章字数预算表
4. 版式观察与需要人工确认的项目
5. 文件命名方案
6. 是否额外生成附件 `.docx`

并等待用户确认或修改。
